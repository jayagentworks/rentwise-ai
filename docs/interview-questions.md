# RentWise AI 面试题与参考回答

## 一、项目介绍

### 1. 请用一分钟介绍这个项目

RentWise AI 是一个面向中国租客的智能租房决策 Agent。它不发布房源，而是接收候选房源，结合预算、真实居住成本、多名家庭成员的高德通勤、硬性条件和个性化偏好完成排序与解释。系统使用 FastAPI 和 LangGraph 编排决策过程，React 提供需求向导和对比界面，PostgreSQL 保存匿名档案、历史和 Agent checkpoint，Redis 缓存高德结果并执行全局 3 QPS 限流。LLM 只负责开放偏好映射和证据约束解释，不直接计算成本、通勤或法律结论。

### 2. 这个项目解决了什么问题？

传统租房平台主要解决“找到房源”，但用户仍需手工比较隐性费用、多人通勤和合同风险。RentWise 位于房源平台之上，解决“哪套房更适合我以及为什么”的决策问题。

### 3. 为什么不直接做一个房源发布平台？

房源供给、审核、即时状态和平台沟通需要大量运营与授权接口。项目把边界放在决策层，选定后跳转原平台联系，既降低集成复杂度，也避免代替用户发送消息和处理平台登录。

### 4. 当前哪些数据是真实的，哪些是模拟的？

高德路线、硅基流动 LLM、合同 OCR 和规则核验是真实调用；房源快照、租金和原平台链接仍是模拟上海数据。系统在界面和文档中明确这一边界，不把模拟房源描述为真实在租房源。

## 二、系统架构

### 5. 为什么选择 React + FastAPI？

React 适合实现多步骤向导、动态家庭目的地、房源对比和中英文状态交互。FastAPI 原生支持异步 I/O，适合并发调用地图、LLM、Redis 和 PostgreSQL，同时自动生成 OpenAPI 文档。

### 6. 为什么使用 LangGraph，而不是普通函数链？

决策过程包含候选搜索、偏好解释、确定性评分、自然语言解释和结果生成等明确阶段。LangGraph让节点边界、运行轨迹、checkpoint 和失败恢复成为显式能力。它不是为了增加“Agent 感”，而是为了让流程可观察、可恢复和可测试。

### 7. LangGraph 有哪些节点？

1. `search_candidates`：从 ListingProvider 获取统一房源模型。
2. `interpret_preferences`：把开放偏好映射到已有标签。
3. `evaluate_and_rank`：强制调用通勤 Skill，并计算成本、约束和得分。
4. `explain_recommendations`：根据结构化证据生成理由和取舍。
5. `finalize_response`：生成稳定 API 响应和假设说明。

### 8. 如何实现失败恢复？

每次搜索先生成 `agent_run_id`，并作为 LangGraph `thread_id`。图的每个节点将纯 JSON 状态写入 PostgreSQL checkpointer。运行失败后，恢复接口验证该运行属于当前匿名用户，再使用相同 thread ID 从最近 checkpoint 继续执行。

### 9. 为什么 checkpoint 状态必须转成纯 JSON？

最初直接保存 Pydantic `Listing` 对象时，PostgreSQL saver 的 msgpack 序列化失败。最终方案是在节点边界将模型转换为 JSON 字典，需要计算时再通过 Pydantic 校验恢复。这样避免启用 pickle 回退，也降低反序列化安全风险。

## 三、通勤与地图

### 10. 通勤时间是如何计算的？

房源坐标和用户地址交给高德 Web 服务，直接使用高德路线规划返回的预计时间和距离，不是仅获取直线距离后自行除以平均速度。开发测试可切换到稳定的 MockMapProvider。

### 11. 为什么高德地图是必须调用的 Skill？

通勤是核心决策证据。如果让 LLM 自主决定是否调用，可能出现跳过路线计算或使用语言模型估计时间的问题。因此 LangGraph 的评分节点确定性调用 CommutePlanningSkill，LLM 无权绕过。

### 12. 如何限制高德请求不超过每秒 3 次？

Redis 中使用 Lua 脚本维护跨实例共享的限流状态。所有后端实例使用同一个限流键，调用前获取许可，因此不是单进程内的简单 sleep。同时对相同地理编码和路线结果缓存，减少重复计费请求。

### 13. 高德不可用时怎么办？

系统返回明确的 503，不会偷偷切换成随机通勤时间。这样可以保持推荐结果的证据可信度。Mock 只在明确的开发配置下启用。

### 14. 多目的地通勤为什么不能只看平均值？

平均值可能掩盖某位家庭成员的极端通勤。系统同时计算每个目的地上限、加权时间、最差时间、每周家庭总时间和成员差距。任何成员超过独立上限都会形成硬条件冲突。

## 四、评分与 LLM

### 15. 推荐分由什么组成？

分数由真实月成本、加权通勤、验证过的偏好标签、硬条件通过奖励和多人通勤公平性惩罚构成。历史反馈在有限范围内加减分，避免无限累积。该分数是内部排序分，不代表概率或房源质量的绝对百分比。

### 16. 哪些条件属于硬条件？

租金上限、真实月成本、卧室数、面积、整租或合租、电梯、养宠、中介费、目标行政区以及每个目的地的通勤上限。

### 17. “真实月成本”如何计算？

月租金加月服务费、物业费、水电估算，再把一次性中介费按租期摊销。首月现金支出单独计算租金、押金、服务费、物业费和中介费，避免把现金流压力与长期月均成本混为一谈。

### 18. LLM 在系统中具体负责什么？

LLM 有三个主要职责：把自定义偏好映射到现有房源标签；根据确定性数字生成自然语言理由；解释合同规则或图片可见现象。它不负责路线时间、费用、最终评分和法律风险等级。

### 19. 如何防止 LLM 幻觉？

提示词只允许使用输入的结构化证据；偏好只能映射到提供的标签；输出使用 JSON schema；温度为 0；解释结果不能修改原始数字。LLM 调用失败时保留确定性模板，不影响核心排序。

### 20. 用户输入“隔音好”但房源没有数据怎么办？

系统会把它标为不可验证偏好，并在结果假设区明确提示未计入评分，而不是让 LLM猜测。若未来 Provider 提供噪音检测或房源结构字段，再把它映射为可验证条件。

### 21. 用户反馈如何影响下一次排序？

点赞产生有限正向加分，点踩、价格过高、通勤太长等反馈产生不同幅度的负向调整。调整限制在固定区间，确保一次或重复反馈不会覆盖硬条件和核心成本通勤证据。

## 五、匿名记忆与数据

### 22. 不登录如何实现长期记忆？

浏览器首次访问生成匿名用户 ID 和随机访问 Token。浏览器保存二者，数据库只保存 Token 的 SHA-256 哈希。后续请求同时携带 ID 和 Token，后端验证后恢复偏好、收藏、历史和反馈。

### 23. 为什么不能只用 localStorage 保存全部偏好？

只用 localStorage 无法支持服务端历史、跨会话反馈学习、Agent 运行关联和数据库分析。当前方案让浏览器只持有匿名凭证，业务数据存 PostgreSQL，同时不要求注册账号。

### 24. 清理浏览器后怎么办？

匿名身份默认无法找回，因此系统提供 JSON 档案导出和导入，可迁移偏好及收藏。它不能达到正式账号的可靠恢复能力，但符合第一版无登录边界。

### 25. 历史记录为什么保存快照？

如果只保存房源 ID，Provider 数据变化或下架后无法还原当时决策。新版历史保存前五个推荐的费用、通勤、评分和原链接快照。旧版记录可按原条件重新计算，但会明确它是当前数据下的新结果。

## 六、合同和图片 Skill

### 26. 合同照片如何核验？

照片或扫描 PDF 先经过视觉 OCR，提取文本后执行确定性规则。规则触发风险等级和法律来源，再由 LLM做通俗解释。数据库只保存文件哈希和报告，不保存原图与正文。

### 27. 为什么不能让 LLM 直接判断合同违法？

法律判断需要现行规则、适用地区和具体事实。系统把“规则匹配”和“语言解释”分开：规则决定风险等级，来源记录生效日、地区和核验日，LLM不能改变等级或宣布最终无效。

### 28. 不同城市的合同规则如何处理？

全国规则始终适用；上海地方规则仅在上海报告中启用。其他城市若尚未加载地方规则，会明确提示人工结合当地规定复核，避免把上海条例错误应用到北京等地区。

### 29. 房源图片分析可以判断什么？

只能报告照片中可见的采光状态、装修磨损、收纳、疑似潮湿痕迹和视角限制，并输出置信度及线下核验标记。不能仅凭图片证明朝向、面积、结构安全、采光时长或当前状态。

### 30. MinIO 为什么默认关闭？

合同和房源图片可能包含隐私。系统默认不持久化原文件，只有管理员开启对象存储并且请求明确携带 `consent=true` 才能保存，同时提供按匿名用户隔离的删除接口。

## 七、测试、部署与安全

### 31. 项目有哪些测试？

Pytest 覆盖评分、成本、通勤 Skill、LLM证据边界、合同规则、图片分析和对象存储授权；Playwright 覆盖匿名恢复、历史详情、合同页面和中英文切换；固定 Agent 评估集检查硬条件、指标边界和不可验证偏好。

### 32. Agent 评估和普通单元测试有什么区别？

单元测试验证具体函数或规则，Agent 评估验证完整输入到排序结果的行为属性，例如顶部房源是否通过硬条件、指标是否有效、不可验证偏好是否被排除。评估结果同时生成 JSON 和 Markdown 报告。

### 33. 数据库如何升级？

容器启动前执行 `alembic upgrade head`。数据库版本记录在 `alembic_version` 表，避免依赖启动时隐式 create_all。当前基线版本为 `20260713_0001`。

### 34. 生产环境做了哪些加固？

生产 Compose 使用只读文件系统、临时目录、资源限制、健康检查和自动重启；API 设置 CORS 白名单、安全响应头和 no-store；Redis 开启 AOF；PostgreSQL 提供备份脚本；密钥只通过环境变量注入。

### 35. 项目目前最大的限制是什么？

房源 Provider 仍是模拟数据，因此系统已经能真实计算“给定房源到目的地的通勤”，但不能证明房源实时存在、租金准确或链接有效。下一步需要合法授权的中国房源数据源。

## 八、压力追问

### 36. 为什么这个项目算 Agent，而不只是筛选器？

它具有结构化状态、阶段化工具调用、外部地图和模型能力、节点持久化、失败恢复、证据解释和用户反馈闭环。但核心数字保持确定性，避免把“Agent”理解为所有决策都交给 LLM。

### 37. 如果请求量增长，首先会遇到什么瓶颈？

高德 QPS 和 LLM 延迟会先成为瓶颈。解决方向是提高缓存命中率、路线矩阵批处理、后台任务、结果流式返回和按供应商额度扩容。数据库 checkpoint 还需要设置保留期和归档策略。

### 38. 为什么 Redis 不能作为长期记忆？

Redis 在本项目中是可丢弃的缓存和限流协调器。匿名档案、收藏、反馈和历史是事实数据，必须进入 PostgreSQL并参与备份，避免缓存淘汰造成业务数据丢失。

### 39. 如何证明推荐结果可解释？

每个结果都能回溯到请求快照、费用字段、高德路线结果、硬条件失败项、匹配标签、评分组成和 Agent 运行轨迹。LLM解释只是这些证据的自然语言展示。

### 40. 如果重新设计，你会优先改什么？

第一是接入合法真实房源 Provider，并建立快照更新和去重；第二是扩大按城市版本化的法律规则；第三是让高德矩阵调用批处理；第四是增加真实线上指标、checkpoint 清理和更完整的浏览器搜索流程测试。

## 九、建议重点背诵的数字

- 高德全局限流：3 QPS。
- 通勤目的地：最多 4 个。
- 合同照片或文件：最多 12 个，总计 30MB。
- 房源分析图片：最多 10 张，总计 25MB。
- 后端测试：15 项。
- Playwright 核心流程：2 条。
- Agent 评估：3/3 通过。
- Alembic 版本：`20260713_0001`。
